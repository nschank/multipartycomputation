Big-Oh Notation and Runtime

We theoretical computer scientists generally don't much care about how fast a program actually runs in seconds or minutes (or months...). When we talk about runtime, we're actually referring to the amount of time it takes relative to something.

As an example, let's pretend we had a list of \(n\) items and wanted to check whether the first element was some particular value \(x\). If we triple the size of the list, then change the value of \(x\), the amount of time it would take to check this would not change: the runtime is constant relative to both \(n\) and \(x\).

But what if we wanted to check if any value in the list were \(x\)? The simplest way to check would obviously be to check every value one by one. If we triple the length of the list, we can expect that it would take three times as long. This algorithm is linear in \(n\); if we increase \(n\) by some factor, the runtime should increase by about the same factor.

It was for this purpose that computer scientists started using Big-Oh Notation, which is a way of concisely describing the runtime of an algorithm. There are five main functions used within Big-Oh Notation: \(o,O,\Theta,\Omega,\omega\) (read little-oh, big-oh, big-theta, big-omega, little-omega).
